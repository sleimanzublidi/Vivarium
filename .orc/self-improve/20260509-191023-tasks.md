### Keep rejected socket listeners from deleting the live app socket
**Source:** Engineering Ideas #1
**Priority:** P0
**Value:** 5
**Feasibility:** 5
**Safety:** 4
**Composite:** 19 (2 × 5 + 5 + 4)
**Description:** Fix socket ownership so a `SocketServer` instance that fails `start()` because another Vivarium process is already listening cannot later unlink that live process's `~/.vivarium/sock`. Implement by retaining the server only after successful start and/or making `stop()` unlink only sockets this instance actually bound.
**Rationale:** The existing live-listener guard prevents immediate socket stealing, but the shutdown path can still remove the primary app's socket and make future hook events disappear until Vivarium restarts. This is a direct user-visible reliability failure in the core hook pipeline with a compact, testable fix.
**Implementation notes:** Prefer the defense-in-depth version: make `SocketServer.stop()` remove the socket only if `listenFD >= 0`, and update `AppDelegate.startSocketServer` to keep a local server until `start()` succeeds before assigning `self.server`. This preserves existing startup behavior while removing the stale failed-instance ownership bug.
**Validation notes:** Extend `SocketServerTests.test_secondStart_throwsWhilePrimaryListening` to call `secondary.stop()` after the failed start and then prove the primary listener remains reachable by connecting to the socket path.
