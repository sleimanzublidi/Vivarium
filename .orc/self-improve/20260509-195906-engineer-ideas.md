# 20260509-195906 — Engineering Ideas

## 1. Prevent failed secondary launches from unlinking the live socket
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 18
**Description:** Make socket ownership explicit so an app instance that fails `SocketServer.start()` never removes `~/.vivarium/sock` on termination. `AppDelegate.startSocketServer(store:normalizer:)` assigns `server` before `try server.start()` succeeds (`Sources/Vivarium/AppDelegate.swift:146-160`), while `SocketServer.stop()` unconditionally removes `socketURL` even when `listenFD` is already `-1` (`Sources/Vivarium/Transport/SocketServer.swift:70-78`). Because `SocketServer.start()` intentionally throws `EADDRINUSE` when another Vivarium process is already listening (`SocketServer.swift:20-29`), a failed second launch can retain a non-owning `server` and later unlink the first launch's live socket.
**Rationale:** Users can hit this by double-launching the app, running from Xcode while a packaged app is open, or starting a test/debug host against the real socket. The existing test proves a rejected second start does not immediately steal the primary listener (`Sources/VivariumTests/Transport/SocketServerTests.swift:48-78`), but it does not cover `stop()`/app termination after that failed start. A safe implementation can assign the AppDelegate property only after successful start and/or have `SocketServer` track whether it bound the path before unlinking; validation should add a regression test that a rejected secondary `stop()` leaves the primary listener reachable.

## 2. Reject pet-pack symlinks that escape the pack root
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** Harden pet-pack validation so extracted zips cannot smuggle symlinks or symlinked spritesheets that point outside the staged pack directory. `PetLibrary.loadPack(at:)` checks that the manifest-selected spritesheet path is a child of the pack directory (`Sources/Vivarium/Pets/PetLibrary.swift:70-85`), and `extractedContentsStayInside(_:)` plus `isChild(_:of:)` use `standardizedFileURL.path` string-prefix checks (`PetLibrary.swift:259-269`, `PetLibrary.swift:315-319`). Those checks do not resolve symlinks, so a zip can place a symlink under the pack root and have `NSImage(contentsOf:)` follow it when loading the spritesheet (`PetLibrary.swift:87-90`).
**Rationale:** Pet packs are user-supplied extension content, so validation should guarantee the app only reads and installs files inside the dropped archive's actual pack root. Current tests cover invalid IDs and non-zip installs (`Sources/VivariumTests/Pets/PetLibraryTests.swift:184-213`) but do not exercise symlink escape cases. A focused fix can reject symlink entries during discovery/install or compare `resolvingSymlinksInPath()` paths before decoding/copying; validation should add a fixture whose manifest points at an in-pack symlink to an external image and assert installation fails with a specific invalid-spritesheet/extraction error.

## 3. Preserve per-connection hook event ordering in SocketServer
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** Dispatch NDJSON lines from each client connection in read order instead of launching an independent unstructured task per line. `SocketServer.readLoop(fd:)` parses newline-delimited records sequentially, but then calls `Task { await self.onLine(line) }` for every line (`Sources/Vivarium/Transport/SocketServer.swift:96-110`) on a concurrent read queue (`SocketServer.swift:13`). The normalizer explicitly does not serialize adapter calls (`Sources/Vivarium/Adapters/EventAdapter.swift:8-13`), and the production handler awaits `SessionStore.apply(_:)` asynchronously (`Sources/Vivarium/AppDelegate.swift:146-155`), so bursts from one hook client can be applied out of order.
**Rationale:** Out-of-order lifecycle events can produce visible wrong state: for example, if `SessionEnd` is applied before an earlier `PreToolUse`, the end is a no-op for an unknown session and the later tool event can leniently recreate a running ghost pet. Existing socket and end-to-end tests assert delivery/eventual running state (`Sources/VivariumTests/Transport/SocketServerTests.swift:12-28`, `Sources/VivariumTests/EndToEnd/EndToEndTests.swift:19-35`) but do not prove ordering under bursty multi-line input. A safe implementation can use one ordered async drain per accepted connection, awaiting `onLine` for each parsed line before processing the next, and add a test that sends `SessionStart`, `PreToolUse`, `SessionEnd` in one connection with artificial handler delay and asserts the final store is empty.
