# 20260509-194814 — Engineering Ideas

## 1. Re-arm restored sessions' idle fallback timers
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 18
**Description:** `SessionStore.restore(from:)` rehydrates sessions and emits `.added` events, but it never restarts the per-session idle fallback timer (`Sources/Vivarium/Sessions/SessionStore.swift:593-637`). Live event application does re-arm that timer through `rescheduleIdleTimer(for:)` (`SessionStore.swift:156`, `SessionStore.swift:238`), and the timer is the only path that turns a stale `.running` / `.review` pet back into `.idle` after `agentIdleTimeout` (`SessionStore.swift:526-551`). Add restore-time timer scheduling for restored non-attention sessions and cover it with a `SessionStoreTests` case that restores a persisted `.running` session, waits past a short `agentIdleTimeout`, and asserts it reaches `.idle`.
**Rationale:** Session persistence is meant to recover visible agent state after app relaunch, but a restored busy pet can remain busy forever if no new hook event arrives. Existing restore tests assert the snapshot round-trip and restored `.running` state (`Sources/VivariumTests/Sessions/SessionStoreTests.swift:554-600`), but they do not verify that restored sessions participate in the same idle fallback lifecycle as live sessions.

## 2. Harden socket framing for final lines and oversized payloads
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** `SocketServer.readLoop(fd:)` appends bytes to an unbounded `Data` buffer until it sees `\n`, launches a task for each newline-terminated line, and immediately returns on EOF (`Sources/Vivarium/Transport/SocketServer.swift:96-111`). That drops a valid final JSON payload if a client closes without a trailing newline, and a malformed local client can grow the buffer indefinitely by never sending one. Add explicit NDJSON framing behavior: process one final non-empty buffer on EOF, cap maximum line size with a logged/drop decision, and test both behaviors alongside the existing small newline-terminated test (`Sources/VivariumTests/Transport/SocketServerTests.swift:12-28`).
**Rationale:** The bundled notifier writes a newline today, but the socket server is the product boundary for hook traffic and should be robust to partial/custom clients rather than silently losing a valid terminal event. This is a transport-framing correctness fix in `SocketServer`, distinct from setup verification or event logging work.

## 3. Make selecting the default sample pet clear stale project overrides
**Value:** 3
**Feasibility:** 5
**Safety:** 4
**Composite:** 15
**Description:** The right-click picker exposes every registered pet from `SceneDirector.availablePets()` (`Sources/Vivarium/Scene/SceneDirector.swift:145-151`), and `AppDelegate.petMenuItemSelected(_:)` calls both `GlobalSettingsStore.setPetID` and the in-memory `SessionStore.setPetID` (`Sources/Vivarium/AppDelegate.swift:410-418`). But `GlobalSettingsStore.setPetID` returns immediately for the bundled `sample-pet` (`Sources/Vivarium/Sessions/ProjectResolver.swift:173-185`), while resolution filters `sample-pet` out of persistable choices (`ProjectResolver.swift:155-170`, `ProjectResolver.swift:220-227`). Change the explicit `sample-pet` selection semantics to remove any existing project-agent mapping, so future resolves fall back to the configured default instead of keeping the old persisted pet. Add a focused test next to the current non-persistence assertions (`Sources/VivariumTests/Sessions/ProjectResolverTests.swift:340-360`).
**Rationale:** Today a user can choose the sample pet from the visible menu and see the current session update, but the choice is not honored for future sessions if that project previously had another persisted pet. Clearing the override preserves the intentional "do not write sample-pet as a mapping" rule while making the menu action durable and unsurprising.
