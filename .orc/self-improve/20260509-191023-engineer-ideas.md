# 20260509-191023 — Engineering Ideas

## 1. Keep rejected socket listeners from deleting the live app socket
**Value:** 5
**Feasibility:** 5
**Safety:** 4
**Composite:** 19
**Description:** Fix the socket ownership path so a `SocketServer` instance that fails `start()` because another Vivarium process is already listening cannot later unlink that live process's `~/.vivarium/sock`. `AppDelegate.startSocketServer(store:normalizer:)` assigns `server = try SocketServer(...)` before `try server.start()` (`Sources/Vivarium/AppDelegate.swift:144-158`), while `SocketServer.stop()` removes `socketURL` even when `listenFD` was never opened (`Sources/Vivarium/Transport/SocketServer.swift:70-78`). A rejected second launch can therefore keep a failed `server` reference and remove the primary app's socket during termination. Implement by retaining the server only after successful start and/or making `stop()` unlink only sockets this instance actually bound. Add a regression test extending `test_secondStart_throwsWhilePrimaryListening` (`Sources/VivariumTests/Transport/SocketServerTests.swift:48-78`) to call `secondary.stop()` after the failed start and prove the primary listener remains reachable.
**Rationale:** The existing live-listener guard correctly prevents immediate socket stealing, but the shutdown path can still strand the real app with no filesystem socket, making future hook events disappear until Vivarium restarts. This is a user-visible reliability bug with a small, well-contained fix.

## 2. Use Copilot hook PID when synthesizing fallback session keys
**Value:** 4
**Feasibility:** 5
**Safety:** 4
**Composite:** 17
**Description:** Include `Envelope.pid` in Copilot fallback session identity, not just `(cwd, ppid, timestamp)`. `CopilotCLIAdapter.Envelope` already decodes both `pid` and `ppid` (`Sources/Vivarium/Adapters/CopilotCLIAdapter.swift:16-20`), but `originKey(cwd:ppid:)` and `synthesizeKey(cwd:ppid:timestamp:)` ignore `pid` (`CopilotCLIAdapter.swift:120-131`). When `payload.sessionId` is absent, concurrent Copilot hook children from the same working directory and parent can collide, especially for out-of-order events that fall back to the `"unknown"` timestamp path (`CopilotCLIAdapter.swift:146-171`). Add adapter tests beside the existing synthetic-key coverage (`Sources/VivariumTests/Adapters/CopilotCLIAdapterTests.swift:28-45`) showing two payloads with the same cwd/ppid but different pid produce distinct session keys while same pid remains stable.
**Rationale:** Session-key collisions merge distinct active agents into one pet, causing lost state transitions and misleading balloons. The fix uses data already present in the hook envelope and is limited to Copilot's legacy fallback path; modern `sessionId` handling remains authoritative.

## 3. Reject pet packs whose extracted contents resolve outside the pack root
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** Harden pet ZIP validation against symlink and alias escapes. `loadPack(at:)` checks that the manifest-selected spritesheet path is syntactically under the pack directory (`Sources/Vivarium/Pets/PetLibrary.swift:70-85`), and `extractZip(_:to:)` calls `extractedContentsStayInside(_:)` after `ditto` extraction (`PetLibrary.swift:206-231`). However, both `extractedContentsStayInside(_:)` and `isChild(_:of:)` compare standardized path strings (`PetLibrary.swift:259-270`, `315-319`) rather than resolving symlink destinations or rejecting symlink entries, so a crafted archive can leave a pack that points outside its installed directory. Add validation that rejects symlinks/aliases or resolves resource values before accepting/copying, plus install tests near `PetLibraryTests.test_installPack_fromFolderZipCopiesIntoUserPets` and unsafe-ID coverage (`Sources/VivariumTests/Pets/PetLibraryTests.swift:153-203`) using a zip with a symlinked spritesheet or nested directory escape.
**Rationale:** Drag-and-drop pet installation is a user-facing file ingestion path. A malicious or malformed pack should not make Vivarium read or persist references to files outside `~/.vivarium/pets/<id>`, and this hardening reduces both security risk and future debugging ambiguity around broken packs.
