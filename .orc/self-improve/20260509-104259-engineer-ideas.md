## 1. Wire up the attention-alert coordinator behind a notifications toggle
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** Activate the attention-alert path that is already implemented and tested but currently commented out at `Sources/Vivarium/AppDelegate.swift:82-93`. Concrete steps:
1. Extend `GlobalSettingsStore.Settings` (`Sources/Vivarium/Sessions/ProjectResolver.swift:94`) with a new field `notificationsEnabled: Bool` (default `false`); bump `version` and provide a forward-compatible decode that tolerates absence of the field. Add a `setNotificationsEnabled(_:)` method that writes through atomically, paralleling the existing `projectPets` API.
2. Uncomment the three coordinator/notifier lines and the `alertCoordinator.handle(event)` call in `applicationDidFinishLaunching`. Wrap the `handle(event)` call in a check on `settingsStore.snapshot().notificationsEnabled` so the path is dormant by default — `SystemSessionAlertNotifier.requestAuthorization()` should only be called the first time the toggle flips on.
3. Add a single "Send notifications when an agent needs attention" menu item to the existing status menu (sibling of Show/Hide Tank), bound to the new setting. No SwiftUI panel required.
4. Add unit tests: (a) coordinator still fires only on edge transitions when enabled; (b) wiring layer suppresses `notifier.notify` calls when `notificationsEnabled == false` using the existing recording double from `SessionAlertCoordinatorTests`; (c) `GlobalSettingsStore` round-trips the new field and decodes a v1 file that lacks it.
**Rationale:** The attention-alert subsystem (`SessionAlertCoordinator` 64 LOC + `SystemSessionAlertNotifier` 69 LOC) is fully built, has a clean `SessionAlertNotifier` protocol seam for tests, and is already covered by the test suite — but it is dead code in production because `AppDelegate` never instantiates it. The TODO comment ("see notifications setting") is the only blocker. This converts a passive UI ("I have to look at the tank to know my agent is waiting") into a proactive one with a single targeted change. Edge-detection in the coordinator already prevents balloon TTL churn from re-firing alerts, so noise risk is bounded by design.
**Notes:** Default-off matters: enabling notifications on first launch would surface a system permission prompt the user did not ask for. Gating behind a menu toggle that the user explicitly flips on is both safer and matches macOS conventions. Touching `AppDelegate` and `GlobalSettingsStore` is the main risk surface; both are well-tested. Keep the menu item label concise so the existing menu layout (Hooks status, Active sessions, Show/Hide Tank, Quit) stays compact.

## 2. Surface pack validation issues in the menu bar
**Value:** 3
**Feasibility:** 5
**Safety:** 5
**Composite:** 16
**Description:** Capture the `outcome.issues: [DiscoveryIssue]` already returned by `PetLibrary.discoverAll` (currently dropped at `Sources/Vivarium/AppDelegate.swift:54`) and surface it in a new "Pets" submenu on the existing `NSStatusItem`. The submenu has two items: `Issues (N)…` (visible only when `N > 0`) and `Open ~/.vivarium/pets`. Clicking `Issues (N)…` opens an `NSAlert` listing each problematic directory plus a humanized description of its `PetLibrary.PetIssue` value (`missingManifest`, `invalidManifest`, `missingSpritesheet`, `invalidDimensions(w, h)`, `duplicateID(id)`). Reuse the existing enum — add a `var humanized: String` extension covered by tests. Scope deferred for this run: drag-and-drop install does not need to update the issues array on the fly; users can re-open the alert after the next launch.
**Rationale:** Pack failures are silent today: rejected packs are logged to the console (which a typical user never opens) and the tank silently continues with the bundled `sample-pet`. A user who drops a `.zip` and sees nothing has no way to tell whether the file was malformed, dimensions were wrong, the id collided, or the drop wasn't seen at all. The data is already collected — only the surface is missing. Explicit roadmap item ("Pets → Issues (N)").
**Notes:** Refines the backlog IDEA-001 by deliberately deferring live drag-and-drop integration to keep the change small and the diff additive. The agent should still verify with `bash Scripts/validate.sh` that the new `humanized` extension and a `discoverAll` round-trip test do not regress the 222-test baseline. Static menu rebuild on launch is sufficient.

## 3. Post-install bridge selftest in `notify` helper and `setup.sh`
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** Add a `--selftest` flag to the `VivariumNotify` helper that wraps a sentinel envelope (`{"agent":"selftest","event":"heartbeat","sessionKey":"selftest","cwd":"…","payload":{}}`) and writes it to `~/.vivarium/sock` using the existing 200 ms connect/write timeouts; have the app's `EventAdapter` recognize the sentinel by exact-match on `agent == "selftest"` and short-circuit it before any `SessionStore` interaction, logging a single `selftest received` line via `OSLog`. Then extend `Scripts/setup.sh` to: (a) check whether `~/.vivarium/sock` exists after the hook merge, (b) if it does, invoke `~/.vivarium/notify --selftest` and report `Bridge: OK` or `Bridge: not reachable`, (c) if it does not, instruct the user to launch `Vivarium.app` and rerun. Add unit tests for sentinel detection in the adapter dispatch plus an XCTest that the selftest envelope round-trips through `SocketServer` end-to-end without creating a session.
**Rationale:** Today the only way a user knows the install worked is to start a real `claude`/`copilot` session and watch for a pet — slow and error-prone. A selftest closes the silent-failure gap with a single command and doubles as a regression-prevention harness for the IPC pipeline that future autonomous changes can lean on.
**Notes:** Carried forward from backlog IDEA-002. The sentinel must be an exact match (no prefix matching, no fuzzy `contains`) to avoid colliding with future agents whose names happen to start with `self*`. Keep the short-circuit branch first in the dispatch chain so the production hot path is unchanged for normal events. `Scripts/setup.sh` already takes `*.vivarium.bak` backups before merging hook entries, so the additional selftest invocation is read-only against the user's environment and does not require a new backup step.
