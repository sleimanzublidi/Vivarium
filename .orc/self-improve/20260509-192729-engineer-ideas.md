# 20260509-192729 — Engineering Ideas

## 1. Re-arm restored sessions' idle fallback timers
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 2 × Value + Feasibility + Safety = 18
**Description:** Restore should re-arm the same `agentIdleTimeout` fallback used for live events. `SessionStore.apply(_:)` eventually calls `rescheduleIdleTimer(for:)` for direct live events (`Sources/Vivarium/Sessions/SessionStore.swift:125-141`, `526-533`), but `restore(from:)` only inserts persisted sessions and emits `.added` (`SessionStore.swift:590-636`). A restored `.running` or `.review` session can therefore stay visually busy until the 600s stale-session eviction rather than returning to idle after the default 30s agent-idle timeout. Add restore-time idle scheduling for restored non-attention sessions, preserving the existing `.waiting` / `.failed` guard in `needsAttention(_:)` (`SessionStore.swift:541-556`). Cover it with a short-timeout async restore test next to `test_snapshotRoundTrip_restoreEmitsAddedSession` (`Sources/VivariumTests/Sessions/SessionStoreTests.swift:554-600`).
**Rationale:** Users who restart Vivarium while an agent was running can see a pet stuck in "working" state for up to ten minutes even if no new agent event arrives. The fix is narrow, follows existing timer behavior, and directly improves correctness of session restoration.

## 2. Quarantine corrupt global settings before writing defaults
**Value:** 3
**Feasibility:** 5
**Safety:** 5
**Composite:** 2 × Value + Feasibility + Safety = 16
**Description:** `GlobalSettingsStore.loadSettings()` logs decode/read failures and returns fresh defaults (`Sources/Vivarium/Sessions/ProjectResolver.swift:252-260`), while later `saveSettings(_:)` writes those defaults or a new mutation back to the same path (`ProjectResolver.swift:263-270`). A corrupt `~/.vivarium/settings.json` can therefore be silently overwritten on the next project-pet resolution or opacity change, losing project pet mappings and window opacity. Mirror the safer `SessionStore` corrupt-snapshot path (`Sources/Vivarium/Sessions/SessionStore.swift:597-604`, `688-702`) by moving unreadable settings to `settings.json.corrupt-<timestamp>` before returning defaults. Add tests in `ProjectResolverTests` covering corrupt settings during `petID(for:)` and `setWindowOpacity(_:)`.
**Rationale:** Settings are user customization state, not disposable cache. Quarantining preserves recoverability, aligns behavior with session snapshot handling, and reduces the chance that a bad write or manual edit permanently erases pet assignments.

## 3. Move dropped pet-pack installation off the main UI path
**Value:** 3
**Feasibility:** 4
**Safety:** 4
**Composite:** 2 × Value + Feasibility + Safety = 14
**Description:** Drag/drop installation currently runs synchronously from the AppKit drag callback: `PetDropSKView.performDragOperation(_:)` invokes `onPetZipDropped` directly (`Sources/Vivarium/Window/FloatingTank.swift:357-360`), `AppDelegate.installDroppedPetZip(_:)` performs the install inline (`Sources/Vivarium/AppDelegate.swift:490-510`), and `PetLibrary.extractZip(_:to:)` launches `/usr/bin/ditto` then blocks with `waitUntilExit()` (`Sources/Vivarium/Pets/PetLibrary.swift:206-217`). Large or slow zip extraction can freeze the tank/menu bar for the full archive extraction and image decode. Introduce an async install coordinator that performs extraction/validation off the main actor, then hops back to register the pack, show the preview, or present the failure alert. Validate with a test that injects a slow installer and asserts the drag/drop handler returns promptly, plus a measured large-zip check targeting UI handoff under 100 ms instead of blocking for the current `ditto` duration.
**Rationale:** Pet packs are a user-facing extension point, and a frozen menu-bar app makes installation feel broken. The change isolates slow filesystem/process work away from UI code while keeping existing `PetLibrary` validation behavior and install failure presentation intact.
