## Selected Task
**Decision:** IMPLEMENT
### Re-arm restored sessions' idle fallback timers
**Source:** Engineering Ideas #1
**Priority:** P1
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 18
**Description:** `SessionStore.restore(from:)` rehydrates sessions and emits `.added`, but never restarts the per-session idle fallback timer. Live event application arms that timer through `rescheduleIdleTimer(for:)`; without the same restore-time scheduling, a restored busy pet can remain `.running` or `.review` indefinitely after app relaunch if no new event arrives.
**Rationale:** The spec explicitly uses the agent-idle timeout to recover from missed done signals and crashed/mid-session agents. Restore currently bypasses that lifecycle, which makes persisted session recovery visually unreliable in the exact stale-session case the fallback exists to handle. The fix is narrow, user-visible, and covered by existing `SessionStoreTests` patterns.
**Implementation notes:** Schedule the idle fallback for restored sessions after they are inserted, preserving the existing attention-state behavior in `idleTimerFired(for:)` so `.waiting` and `.failed` sessions are not auto-cleared. Avoid changing snapshot decode, stale-session filtering, or live event application semantics.
**Validation notes:** Add a focused restore test that persists a `.running` session, restores it into a store with a short `agentIdleTimeout`, waits past the timeout, and asserts the session transitions through the existing fallback to `.idle`. Keep existing snapshot round-trip and stale-restore tests passing.
