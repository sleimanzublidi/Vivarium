## Debate Summary

Three candidates landed at composite 16 in the upstream files: bridge selftest (carries `IDEA-002`), re-enable attention alerts (new), and surface pack validation issues (carries `IDEA-001`). Backlog also offers `IDEA-003` (persist SessionStore) and `IDEA-004` (first-run onboarding, strategic).

Adversarial review changed the picture:

- **Attention alerts (new).** I bumped Safety to 5. The production paths (`SessionAlertCoordinator` 64 LOC, `SystemSessionAlertNotifier` 69 LOC) are already implemented and covered by `SessionAlertCoordinatorTests` (151 LOC, edge-detection cases). The actual diff is: uncomment 3 lines at `AppDelegate.swift:82-93`, gate the `alertCoordinator.handle(event)` call on a new `notificationsEnabled` flag in `GlobalSettingsStore.Settings` (which already uses `decodeIfPresent` for forward compatibility — see `ProjectResolver.swift:103-107`), add one menu item, and request authorization only on first toggle-on. Default-off means existing users see no behavior change and the OS won't surface an unsolicited permission prompt. → V=4, F=4, S=5, **C=17**.

- **Bridge selftest (`IDEA-002`).** I dropped Feasibility and Safety to 3. The writeups specify "setup.sh runs `~/.vivarium/notify --selftest` and reports `Bridge: OK` or `Bridge: not reachable`" but do not specify how `setup.sh` actually verifies the sentinel was received. `notify` always exits 0 today; the OSLog-based "selftest received" line isn't readable from `setup.sh` without grant flow. A real implementation needs either a non-zero exit on selftest write failure (deviation from notify's "always exit 0" contract) or a marker-file roundtrip — neither is sketched. Plus four moving parts (helper, dispatcher, setup.sh, tests) and `setup.sh` modifications touch the user's `~/.claude/settings.json` indirectly. Implementable, but not in one safe autonomous run as currently scoped. → V=4, F=3, S=3, **C=13**.

- **Pack validation (`IDEA-001`).** Existing drag-drop failures already raise `presentInstallFailure` `NSAlert` (`AppDelegate.swift:410-417`); the *new* coverage is the discovery-time issues from `discoverAll` (`AppDelegate.swift:54`) — narrower than the writeup implies. Genuinely safe and additive; deliberately ships static menu rebuild on launch. → V=3, F=5, S=5, **C=16**.

Strategic items (`IDEA-004`) and `IDEA-003` (persist SessionStore — modifies the actor at the heart of the app, F=3, S=3) are not stronger than attention alerts for this run.

**Selection: re-enable attention alerts.** Highest composite, lowest regression risk per LOC touched, and most user-facing per session (every `.waiting`/`.failed` edge in any project for any agent).

## Rejected Ideas

### Bridge selftest (Product #1 / Engineering #3 / Backlog `IDEA-002`)
**Reason:** Composite drops to 13 after adversarial review. The verification mechanism by which `setup.sh` distinguishes "Bridge: OK" from "Bridge: not reachable" is not specified — `notify` exits 0 unconditionally today, and OSLog-based detection is unreliable from a shell script. Implementing this safely requires a design decision (exit-code semantics for `--selftest`, or a marker-file roundtrip with timeout) plus changes across four components (`VivariumNotify/main.swift`, `EventNormalizer`/`EventAdapter` dispatch, `Scripts/setup.sh`, tests). Stays on the backlog as `IDEA-002` for a future run that owns the design decision.

### Surface pack validation issues (Product #3 / Engineering #2 / Backlog `IDEA-001`)
**Reason:** Genuinely safe and small (C=16), but value is bounded: drag-drop install failures already raise an `NSAlert` (`AppDelegate.swift:410-417`), so the new coverage is limited to discovery-time issues from `~/.vivarium/pets/<id>` directories that exist on disk and fail validation, plus duplicate-id collisions across runs. Lower per-session impact than attention alerts. Stays on the backlog as `IDEA-001`.

### Persist SessionStore across restarts (Backlog `IDEA-003`)
**Reason:** Modifies the actor at the heart of the app (F=3, S=3). Restore-time event ordering and debounced-write actor reasoning need careful design; not a one-run autonomous candidate. Stays on backlog.

### First-run onboarding window (Backlog `IDEA-004`)
**Reason:** Strategic, F=2. Read-only detection slice has shipped (`HookInstallationProbe` + active-sessions submenu). Remaining write-path requires the bridge selftest to land first as a verifier. Kept strategic.

## Selected Task
**Decision:** IMPLEMENT
### Wire up the attention-alert coordinator behind a notifications toggle
**Source:** Engineering Ideas #1 (also Product Ideas #2; new this run, not previously in backlog)
**Priority:** P1
**Value:** 4
**Feasibility:** 4
**Safety:** 5
**Composite:** 2 × 4 + 4 + 5 = 17

**Description:** Activate the dead-coded attention-alert path at `Sources/Vivarium/AppDelegate.swift:82-93`, gated by a new `notificationsEnabled: Bool` field on `GlobalSettingsStore.Settings` (default `false`). Add a single status-menu toggle that flips the setting and, on first toggle-on, calls `SystemSessionAlertNotifier.requestAuthorization()`. When enabled, the existing `SessionAlertCoordinator` posts one banner per edge into `.waiting` (with sound) or `.failed` (silent).

**Rationale:** Today an agent that asks the user a question stalls invisibly until the user happens to look at the tank. The notification subsystem is already implemented (`SessionAlertCoordinator` 64 LOC + `SystemSessionAlertNotifier` 69 LOC), already has a clean `SessionAlertNotifier` protocol seam, and is already covered by `SessionAlertCoordinatorTests` (151 LOC including the bouncing-state edge-detection cases). The only blocker is the toggle UI plus the wiring at `applicationDidFinishLaunching` — exactly the surface this run can land safely. Converts Vivarium from passive ("look at the tank to see if your agent needs you") to active ("you'll know") with a single targeted change. Default-off is non-negotiable on macOS (no unsolicited permission prompts), and edge-detection in the coordinator already bounds noise.

**Implementation notes:**
1. **`GlobalSettingsStore.Settings`** (`Sources/Vivarium/Sessions/ProjectResolver.swift:94`):
   - Add `var notificationsEnabled: Bool` (default `false`) to the struct, the memberwise init, and `init(from decoder:)` using `decodeIfPresent` so existing `settings.json` files without the field decode cleanly. Existing `version: 1` stays — the field is purely additive and the decode is forward-compatible.
   - Add `func notificationsEnabled() -> Bool` and `func setNotificationsEnabled(_ enabled: Bool)` paralleling the existing `petID(...)` / `setPetID(...)` API. Both must take the existing `lock`. `set...` calls `saveSettings(...)` only when the value actually changes.
2. **`AppDelegate`** (`Sources/Vivarium/AppDelegate.swift`):
   - Uncomment the three lines at 82-93 (`alertNotifier`, `alertCoordinator`, `handle(event)`). Keep them as ivars; do **not** call `requestAuthorization()` from `applicationDidFinishLaunching` — only call it when the user flips the toggle to ON for the first time.
   - Wrap `alertCoordinator.handle(event)` in `if settingsStore.notificationsEnabled() { alertCoordinator.handle(event) }`. Read on each event so toggle changes take effect immediately without a relaunch. (`GlobalSettingsStore.loadSettings` reads from disk on each call, which matches the existing `petID(for:...)` pattern; this is the heaviest hot path so a small cache or in-memory cached value is acceptable if profiling later shows it matters — out of scope for this run.)
   - Add a new menu item to `rebuildStatusItemMenu` between "Active sessions" and "Show / Hide Tank": title `"Notify on agent attention"`, `state` reflecting `notificationsEnabled()`, action selector that flips the setting and, when transitioning `false → true` *and* this is the first time the toggle has ever flipped on, calls `alertNotifier.requestAuthorization()`. Track first-flip-on with a transient `Bool` ivar on `AppDelegate` (no need to persist — `requestAuthorization` is idempotent at the OS layer; persisting "have we asked" only avoids a single redundant call).
   - The menu rebuilds on every open (`menuNeedsUpdate`), so the checkmark stays in sync with the setting without manual refresh.
3. **Tests** (`Sources/VivariumTests/Sessions/`):
   - `ProjectResolverTests` (or a new `GlobalSettingsStoreTests` file if cleaner): round-trip `notificationsEnabled` through save/load; decode an existing-shape v1 file (without the field) and assert it returns `false`; flipping `setNotificationsEnabled(true)` then reloading returns `true`.
   - `SessionAlertCoordinatorTests`: the existing tests cover the coordinator, which does **not** change. The wiring-layer suppression (gate skipping `handle(event)` when disabled) is best covered by a small AppDelegate-level test if practical, otherwise verified by manual smoke (acceptable; the gate is a one-line `if`).
4. **No changes** to `SessionAlertCoordinator`, `SystemSessionAlertNotifier`, the `SessionAlertNotifier` protocol, or any adapter/dispatcher code. The selected scope is purely wiring + 1 settings field + 1 menu item + 1 gate.

**Validation notes:**

- Add the new `GlobalSettingsStore` round-trip tests; they should pass on the first run.
- Manual smoke (document only — autonomous run can't grant macOS notification permission): with the toggle ON, drive a `.waiting` event into the running app and confirm a banner; toggle OFF, drive another `.waiting`, confirm no banner. This step is not blocking for the autonomous run — code-level coverage via the existing protocol seam is sufficient.
- Confirm that `~/.vivarium/settings.json` produced by an existing build (no `notificationsEnabled` key) decodes without error after the field is added (existing `decodeIfPresent` pattern in `Settings.init(from:)` is the load-bearing piece — verify in test).
