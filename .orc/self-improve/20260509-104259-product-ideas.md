# 20260509-104259 — Product Ideas

## 1. Bridge selftest after install so users know hooks actually work
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** After running `./Scripts/setup.sh --claude` (or `--copilot` / `--both`), users currently have no way to confirm the install worked except to start a real `claude` or `copilot` session and watch the tank for a pet that may or may not appear. Add a `--selftest` mode to the `VivariumNotify` helper that emits a sentinel envelope (`agent: "selftest"`, `event: "heartbeat"`) onto `~/.vivarium/sock`. Have `Vivarium.app` recognize the sentinel in its event normalizer and log a single "selftest received" line without touching `SessionStore`. Then have `Scripts/setup.sh`, immediately after merging hooks, run the selftest and print one of three explicit verdicts: `Bridge: OK`, `Bridge: not reachable — launch Vivarium.app and rerun`, or `Bridge: app installed but socket missing — launch Vivarium.app first`. The user finishes setup with a confident green check or a pointed instruction, never a silent failure.
**Rationale:** This is the single highest-friction moment in the product. The repo summary, README, and `IDEA-002` all converge on the same observation: a fresh user runs the script, sees no errors, and then has to invent a way to test it. Every minute of "did it work?" uncertainty at install time is a minute the user is more likely to give up. The hook installer is the funnel; everything downstream (pets, alerts, packs) only matters if the bridge is wired. A one-line verdict in the same script the user already runs converts the riskiest step into a self-verifying one. Refines backlog `IDEA-002`: framing kept, user-facing copy made explicit, and the success/failure split clarified into three named states so the user always knows the next step.

## 2. Re-enable attention alerts so users don't miss agents waiting for input
**Value:** 4
**Feasibility:** 4
**Safety:** 4
**Composite:** 16
**Description:** When a Claude Code or Copilot CLI session enters `.waiting` (the agent is asking the user a question) or `.failed`, raise a native macOS notification with the project name and the most recent balloon text. The wiring already exists — `SessionAlertCoordinator` and `SystemSessionAlertNotifier` are implemented and unit-tested — but is commented out at `AppDelegate.swift:82-93` pending a user-visible setting. Ship it behind a single menu-bar toggle: under the existing menu add **Notifications: Attention alerts** as a checkmark item (default ON), persisted in `UserDefaults`. On first launch the app calls `requestAuthorization()`; if the user denies macOS notifications the menu item shows "(notifications disabled in System Settings)" and is disabled. Edge-only firing is already correct (one alert per transition into `.waiting` or `.failed`), so no spam.
**Rationale:** Long-running agent sessions routinely stall on a question the user can't see — they switched to another window, the tank is occluded, or they walked away. Today the only signal is the pet's idle pose, which by design is silent. This is the recurring, every-session value-add that turns Vivarium from a passive desktop pet into an active workflow tool. Implementation is unusually low-risk for the impact: the production code path is already tested, only the toggle UI and `applicationDidFinishLaunching` wiring need to land. Not in the existing backlog as a standalone idea — extracted from the gap noted in section 7 of the codebase summary, where the wiring is committed but commented "see notifications setting TODO."

## 3. Surface pack validation issues so failed pet drops aren't silent
**Value:** 3
**Feasibility:** 5
**Safety:** 5
**Composite:** 16
**Description:** A user drops an OpenPets `.zip` onto the tank, nothing visible happens, and they have no way to tell whether the pack was rejected, the drop wasn't seen, or the pet just looks like another one already in their library. The app *already* knows the answer: `PetLibrary.discoverAll` returns `outcome.issues: [PetIssue]` with structured reasons (missing manifest, invalid manifest, missing spritesheet, wrong dimensions, duplicate id), and that array is dropped on the floor at `AppDelegate.swift:54`. Capture it. Add a **Pets** submenu to the existing menu bar status item with two items: an `Issues (N)…` row that's only visible when `N > 0`, and `Open ~/.vivarium/pets` to reveal the folder in Finder. Clicking `Issues (N)…` opens a single `NSAlert` listing each rejected directory with a humanized reason ("`my-pet/sheet.png` is 1500×1800; expected 1536×1872"). Update the count live when a drag-and-drop install fails so a bad zip drop produces an immediate "1 issue — click to view" hint in the menu.
**Rationale:** Pack failures are the second-most-common silent failure in the product after broken hooks. Users who customize their pet library are the engaged users; they're the most disappointed when a drop appears to do nothing. The data is already collected and structured — only the UI surface is missing — which makes this the safest and fastest of the three to implement. Roadmap item with explicit framing in `README.md` ("Pets → Issues (N)") and backlog `IDEA-001`. Carries the backlog idea forward unchanged in scope; refinement is the user-copy specificity ("expected 1536×1872") and the `Open ~/.vivarium/pets` companion item that turns the alert into something the user can actually act on without leaving the menu.
