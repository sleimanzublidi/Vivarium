## Selected Task
**Decision:** IMPLEMENT
### Prevent failed secondary launches from unlinking the live socket
**Source:** Engineering Ideas #1
**Priority:** P0
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 18
**Description:** Make socket ownership explicit so an app instance that fails `SocketServer.start()` never removes `~/.vivarium/sock` on termination. Today, `AppDelegate.startSocketServer(store:normalizer:)` assigns the `server` property before `try server.start()` succeeds, and `SocketServer.stop()` removes `socketURL` even if the instance never bound or listened on that path.
**Rationale:** A user can hit this by double-launching Vivarium, running from Xcode while the packaged app is open, or starting another host against the real socket. Once the failed secondary instance terminates, it can unlink the primary instance's live socket, causing hooks to stop reaching the app while the visible app keeps running. This is a direct reliability break in the core event pipeline, has strong code evidence, and can be fixed with a narrow ownership guard.
**Implementation notes:** Assign `AppDelegate.server` only after successful `SocketServer.start()` and/or teach `SocketServer` to track whether it owns the bound socket path before unlinking it in `stop()`. Prefer the server-side ownership guard because it protects all callers, with the AppDelegate assignment order as an additional cleanup if needed.
**Validation notes:** Add a regression test that starts a primary `SocketServer`, attempts a rejected secondary start on the same socket URL, calls `stop()` on the rejected secondary, and verifies the primary listener remains reachable and can still deliver a line.
