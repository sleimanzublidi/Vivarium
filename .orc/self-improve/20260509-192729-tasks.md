### Re-arm restored sessions' idle fallback timers
**Source:** Engineering Ideas #1
**Priority:** P0
**Value:** 4
**Feasibility:** 5
**Safety:** 5
**Composite:** 18 (2 × Value + Feasibility + Safety, max 20)
**Description:** Update `SessionStore.restore(from:)` so restored sessions in non-attention states get the same `agentIdleTimeout` fallback timer as live sessions. Preserve the existing stale-session cutoff and the `.waiting` / `.failed` attention-state guard so restored sessions that need user action are not auto-cleared.
**Rationale:** Users who restart Vivarium while an agent session is `.running` or `.review` can see that pet remain visually busy until the 10 minute eviction sweep, even though the live-event path would fall back to idle after the default 30 seconds. This violates the documented idle fallback behavior and makes restored state look less trustworthy.
**Implementation notes:** In `SessionStore.restore(from:)`, after inserting each valid restored session, call the existing idle scheduling path for sessions where `!Self.needsAttention(session.state)` and `session.state != .idle`. Do not reschedule for stale records that are dropped, existing sessions skipped due to live-event races, or attention states. Add/adjust tests near `test_snapshotRoundTrip_restoreEmitsAddedSession` using a short `agentIdleTimeout`.
**Validation notes:** Cover restored `.running` or `.review` transitioning through the fallback after the short timeout, and restored `.waiting` / `.failed` remaining untouched. Existing snapshot restore tests should continue to pass.
